# Karpathy Guidelines

Behavioral guidelines inspired by Andrej Karpathy's observations about common LLM coding mistakes.

## Why

LLMs tend to over-engineer, add unnecessary abstractions, and change more code than needed. These guidelines counteract those tendencies with concrete principles.

## Setup

```bash
/plugin install karpathy-guidelines@eneoplugin
```

Guidelines are automatically loaded via CLAUDE.md in every conversation.

## Invoke manually

```
/karpathy-guidelines
```

Use this when you want to explicitly apply the guidelines during a code review, refactor, or implementation.

## Core principles

### 1. Understand before acting

> "The most dangerous thing an LLM can do is confidently write code for a codebase it doesn't understand."

- Read first, write second
- Map dependencies before changing code
- Follow existing patterns
- Ask when uncertain

### 2. Embrace simplicity

> "The best code is no code. The second best code is simple code."

- Solve the actual problem, not adjacent ones
- Avoid abstraction addiction
- Resist feature creep
- Delete dead code boldly

### 3. Make surgical changes

> "Every line you change is a line that could introduce a bug."

- Minimal diff
- Preserve working code
- One concern per change
- Verify assumptions

### 4. Define success clearly

> "If you can't describe what success looks like, you can't achieve it."

- Clarify requirements upfront
- Set measurable criteria
- Surface blockers immediately
- Verify against original goals

## Anti-patterns

| Anti-Pattern | Better Approach |
|--------------|-----------------|
| Refactoring while fixing bugs | Separate commits |
| Adding error handling "just in case" | Handle errors that can actually occur |
| Creating utilities for one-time operations | Inline the code |
| Over-documenting obvious code | Write self-documenting code |
| Guessing at implementation | Read the code or ask |
