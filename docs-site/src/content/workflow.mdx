# Workflow

The harness uses three lanes. Pick the lightest one that still gives enough structure.

## Fast

Use for tiny, bounded changes.

```bash
/eneo-core:eneo-new "<change>"
```

Typical path:

```text
/eneo-new → edit → /eneo-verify → /eneo-commit → /eneo-ship
```

Good for:

- typo fixes
- small docs changes
- one-file implementation fixes

## Standard

Use when the change spans a few files or needs a small spec.

Typical path:

```text
/eneo-new → /eneo-start → /eneo-verify → /eneo-commit → /eneo-ship
```

Good for:

- small feature slices
- bounded backend + frontend changes
- changes that benefit from phase execution but do not need a full PRD

## Deep

Use when the change is risky, cross-cutting, or needs explicit planning.

Typical path:

```text
/eneo-new → /eneo-discuss → /eneo-plan → /eneo-start → /eneo-verify → /eneo-commit → /eneo-ship → /eneo-recap
```

Good for:

- tenancy changes
- audit schema changes
- auth or authz work
- migrations
- large cross-module changes

## What changes for the developer

The common path stays small:

- small work should feel close to “make the edit and verify it”
- bigger work gets more structure only when the risk justifies it

You should not need to remember every rule manually. The standards plugin carries that load through hooks, validators, and helper commands.
