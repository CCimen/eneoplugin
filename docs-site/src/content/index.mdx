# eneo-agent-harness

A Claude Code marketplace for the Eneo workflow.

## TL;DR

- install `eneo-core` and `eneo-standards`
- start with `/eneo-core:eneo-new "<change>"`
- run `/eneo-core:eneo-doctor` when the environment feels wrong
- let `eneo-standards` handle TDD, audit, tenancy, and ratchet enforcement

## What developers get

- one shared way to start work
- a lighter common path for small changes
- a dedicated commit-review step before PR creation
- stronger guardrails for risky or cross-cutting work
- deterministic diagnostics instead of ad hoc shell spelunking
- findings capture without mixing backlog work into the current task

## Marketplace structure

| Plugin | Purpose |
|--------|---------|
| [eneo-core](/plugins/eneo-core) | Main `/eneo-*` workflow |
| [eneo-standards](/plugins/eneo-standards) | Hooks, validators, helper commands, status line |
| [eneo-findings](/plugins/eneo-findings) | Findings workflow and learning extraction |
| [eneo-review](/plugins/eneo-review) | Trigger-gated external review agents |

## Quick start

```bash
/plugin marketplace add CCimen/eneoplugin
/plugin install eneo-core@eneoplugin
/plugin install eneo-standards@eneoplugin
/plugin install eneo-findings@eneoplugin
/plugin install eneo-review@eneoplugin
/reload-plugins
```

Then run:

```bash
/eneo-core:eneo-doctor
```

Then start work with:

```bash
/eneo-core:eneo-new "<what you want to change>"
```

## Why this exists

The harness gives Eneo developers one consistent flow for:

- triage
- phase execution
- review gates
- findings capture
- milestone closeout

It is designed to keep the common path light and only add more process when the work is genuinely risky or large.
